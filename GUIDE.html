Docker Compose + Restic + OneDrive
每個服務自行宣告備份

從一台尚未部署本方案的 Linux 主機開始，完成工具安裝、OneDrive 授權、服務 manifest、首次備份、還原測試與 systemd 自動排程。

1. 目標與架構

服務仍由 Docker Compose 管理。 每個服務在自己的目錄放一份 backup.yaml，自行宣告排程、保留策略、備份來源與一致性方式。
/srv/stacks/minecraft/backup.yaml ─┐ /srv/stacks/palworld/backup.yaml ├─ backupctl 每 5 分鐘掃描 /srv/stacks/ghost/backup.yaml ┘ ↓ 停止服務或執行 hook ↓ 本機 staging ↓ Restic 加密、去重、快照 ↓ rclone → OneDrive repository
元件用途
Docker Compose執行 Minecraft、Palworld、Ghost 等服務。
backup.yaml每個服務自己的排程、保留策略與來源清單。
backupctl掃描 manifest、建立 staging、控制 Compose、執行 Restic。
Restic加密、去重、snapshot、forget、prune、check、restore。
rclone讓 Restic 使用 OneDrive 作為 repository backend。
systemd timer每 5 分鐘執行一次到期檢查；另有週期性維護。

2. 前置條件

  • Debian、Ubuntu 或 Linux Mint 類型系統。
  • 目前登入帳號具備 sudo 權限。
  • 兩台主機需要各自執行本手冊。
  • Microsoft OneDrive 帳號可正常登入。
  • 服務可以短暫停止，或能透過 hook 產生一致性快照。

下列命令用來確認系統：

cat /etc/os-release
uname -a
id
sudo -v

3. 安裝或確認 Docker Compose

先檢查：

docker --version
docker compose version

若兩者都能正常輸出，可以直接前往下一章。

Ubuntu／Linux Mint 安裝 Docker 官方套件

sudo apt update
sudo apt install -y ca-certificates curl

sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

. /etc/os-release

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
https://download.docker.com/linux/ubuntu \
${UBUNTU_CODENAME:-$VERSION_CODENAME} stable" \
| sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update
sudo apt install -y \
  docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin

sudo systemctl enable --now docker

讓目前帳號可操作 Docker

sudo usermod -aG docker "$USER"

登出後重新登入，再測試：

docker run --rm hello-world
docker compose version
加入 docker 群組等同授予高權限。這台電腦只有你使用時較容易接受，但仍不應把不受信任的人加入此群組。

4. 建立 Compose 服務目錄

Compose 專案統一放在：

/srv/stacks/<service>

建立並交給目前使用者：

sudo mkdir -p /srv/stacks
sudo chown -R "$USER":"$USER" /srv/stacks
sudo chmod 755 /srv /srv/stacks

建議結構：

/srv/stacks/
├── minecraft/
│   ├── compose.yaml
│   ├── .env
│   ├── backup.yaml
│   └── data/
├── palworld/
│   ├── compose.yaml
│   ├── .env
│   ├── backup.yaml
│   └── data/
└── ghost/
    ├── compose.yaml
    ├── .env
    ├── backup.yaml
    └── content/

一般權限：

chmod 644 compose.yaml backup.yaml
chmod 600 .env

資料目錄先讓 container 建立或使用預設權限；只有遇到 permission denied 時，才依 image 的 UID/GID 個別調整。

5. 可選：用 Git 保存 /srv/stacks

可以把各服務的 compose.yaml、Dockerfile、Caddyfile、scripts 與其他非機密部署檔保存到 private Git，作為版本管理與額外副本。

這是加分項，不是本專案災難復原的必要條件，也不在本專案主要流程內。即使沒有另外備份 /srv/stacks，仍可從 Restic snapshot 取回服務資料、Compose 檔與自動保存的 backup.yaml。

不要把公開 Git 當成 secrets 儲存位置。真正的 .env、Restic 密碼與 rclone token 不應公開。

7. 設定 OneDrive

使用 root 的獨立 rclone 設定：

sudo rclone config \
  --config /etc/homelab-backup/rclone/rclone.conf

建立 remote 時：

  1. 選擇 n 建立新 remote。
  2. 名稱固定使用 onedrive。
  3. Storage 選 Microsoft OneDrive。
  4. 依提示完成 Microsoft OAuth。
  5. 選擇正確的 OneDrive drive。

完成後：

sudo chmod 600 /etc/homelab-backup/rclone/rclone.conf

sudo rclone lsd onedrive: \
  --config /etc/homelab-backup/rclone/rclone.conf

本方案不使用 rclone mount。Restic 直接使用：

rclone:onedrive:Backups/restic/server-a

8. 設定本機 repository

編輯：

sudo editor /etc/homelab-backup/config.yaml

第一台主機

version: 1
host_id: server-a
services_root: /srv/stacks

repository: rclone:onedrive:Backups/restic/server-a
password_file: /etc/homelab-backup/restic-password
rclone_config: /etc/homelab-backup/rclone/rclone.conf

staging_root: /var/lib/homelab-backup/staging
restore_root: /var/lib/homelab-backup/restores
state_root: /var/lib/homelab-backup/state
cache_root: /var/cache/homelab-backup/restic

lock_file: /run/lock/homelab-backup.lock
volume_helper_image: homelab/volume-rsync:1

rclone:
  bwlimit: ""

check:
  read_data_subset: 5%

第二台主機

只需改成：

host_id: server-b
repository: rclone:onedrive:Backups/restic/server-b
兩台主機可以共用同一個 OneDrive 帳號，但應使用不同 Restic repository。這樣 lock、prune、check 與故障範圍不會互相干擾。

9. 備份災難復原必要設定

完成 OneDrive 與全域設定後，在專案目錄執行：

cd ~/Apps/homelab-backup
sudo ./backup-configs.sh

此腳本只允許 root 執行，因為輸出內容包含 Restic 密碼與 rclone 帳戶憑證。

腳本會把以下三個必要檔案複製到專案的 configs/：

configs/restic-password
configs/rclone.conf
configs/config.yaml

接著只能選一種保存方式：

  1. Git：腳本先顯示 configured remotes；確認所有目的地都是嚴格控管的 private repository 後，必須輸入完整的 PRIVATE，腳本才會提交設定並詢問是否 git push。
  2. ZIP：建立 homelab-backup-configs-YYYYMMDD-HHMMSS.zip。
restic-password 與 rclone.conf 都是 secrets。Git 選項只能用於嚴格控管存取權的 private repository。ZIP 本身沒有加密，應放入加密密碼管理器或離線加密媒體。

從 configs/ 放回設定

cd ~/Apps/homelab-backup
./restore-configs.sh

還原腳本會將檔案安裝到：

/etc/homelab-backup/restic-password
/etc/homelab-backup/rclone/rclone.conf
/etc/homelab-backup/config.yaml

如果 configs/ 缺少任何一個必要檔案，腳本會列出缺失項目、要求使用者先補齊，然後終止，不會執行部分還原。

10. 每個服務的 backup.yaml

每個服務在自己的 Compose 目錄放一份 manifest：

backup.yaml 由備份程式自動保存到每份 snapshot 的 _meta/backup.yaml，不需要再把它寫進 sources.paths。
目錄名稱必須與 service 完全相同；manifest、compose、sources 與各 source entry 的未知欄位會被拒絕，避免拼字錯誤造成靜默漏備份。
/srv/stacks/minecraft/backup.yaml

完整結構：

version: 1
service: minecraft
enabled: true

schedule:
  cron: "0 2,8,14,20 * * *"
  retry_after: 30m
  max_lateness: 2h

retention:
  keep_last: 8
  keep_daily: 14
  keep_weekly: 8
  keep_monthly: 12
  keep_yearly: 3

compose:
  files:
    - compose.yaml

consistency:
  mode: stop
  timeout: 120
  services:
    - minecraft
  before: []
  after: []

sources:
  paths:
    - id: compose
      path: compose.yaml

    - id: env
      path: .env
      required: false

    - id: data
      path: ./data
      exclude:
        - logs/**
        - crash-reports/**

  volumes: []

11. 一致性模式

模式行為用途
stop預同步 → 停止指定 Compose services → 最終同步與 volume 備份 → 重啟原本運行的 services。Minecraft、Palworld、持續寫入檔案、raw database volume。
hooks執行 before hook → 同步 → 執行 after hook；不自動停服。SQL dump、RCON save、應用內建 snapshot。
none直接同步，不停止、不執行一致性 hook。靜態檔、唯讀資料、可重建內容。

stop 範例

consistency:
  mode: stop
  timeout: 180
  services:
    - ghost
    - db

hooks 範例

consistency:
  mode: hooks
  before:
    - ./scripts/dump-database.sh
  after: []
不確定時先使用 stop。它較保守，也比較容易實際驗證還原結果。

12. Bind mount 與 named volume

Bind mount

Compose：

services:
  minecraft:
    volumes:
      - ./data:/data

Manifest：

sources:
  paths:
    - id: data
      path: ./data
      exclude:
        - logs/**
        - crash-reports/**

Compose named volume

Compose：

services:
  db:
    volumes:
      - db-data:/var/lib/mysql

volumes:
  db-data:

Manifest：

sources:
  volumes:
    - id: database
      compose_volume: db-data

compose_volume 填 Compose 頂層邏輯 key，不是實際 Docker volume 名稱。工具會透過 docker compose config --format json 解析。

外部 volume

sources:
  volumes:
    - id: shared-data
      name: company_shared_data

Anonymous volume

以下形式沒有穩定名稱，不適合宣告式備份：

volumes:
  - /data

應改成 named volume：

services:
  app:
    volumes:
      - app-data:/data

volumes:
  app-data:

13. Cron 排程

格式固定為標準 5 欄：

分鐘 小時 日期 月份 星期
表達式意義
0 2 * * *每天 02:00
0 2,8,14,20 * * *每天 02:00、08:00、14:00、20:00
*/15 * * * *每 15 分鐘
0 */6 * * *每 6 小時整點
30 3 * * mon-fri週一到週五 03:30
0 4 1 * *每月 1 日 04:00

排程器每 5 分鐘掃描，因此實際開始時間可能比 cron 時點晚數分鐘。

備份超過 5 分鐘

  • systemd oneshot 不會同時再啟動第二份相同 service。
  • 全域 flock 阻止手動備份、排程、還原與維護重疊。
  • 多個服務同時到期時依序執行，不平行寫入 repository。
  • 執行期間經過的 cron 時點不建立工作佇列。
例如 02:00 開始，備份到 08:20，而 cron 也包含 08:00：08:00 不會另開第二份；完成後下一個有效時點是 14:00。

失敗重試

schedule:
  cron: "0 */6 * * *"
  retry_after: 30m
  max_lateness: 2h

失敗後至少等待 30 分鐘再重試；若該 cron 時點已遲到超過 2 小時，放棄該次，等待下一時點。

14. 每個服務的保留策略

retention:
  keep_last: 8
  keep_daily: 14
  keep_weekly: 8
  keep_monthly: 12
  keep_yearly: 3
欄位意義
keep_last保留最新 N 份。
keep_hourly最近 N 個有 snapshot 的小時，各留一份。
keep_daily最近 N 個有 snapshot 的日期，各留一份。
keep_weekly最近 N 個有 snapshot 的週，各留一份。
keep_monthly最近 N 個有 snapshot 的月份，各留一份。
keep_yearly最近 N 個有 snapshot 的年份，各留一份。
多個 keep-* 條件是聯集，不是總數相加，也不是絕對最大數量。只要符合任一規則就保留。若要嚴格只留最新 30 份，只設定 keep_last: 30。

15. 服務範例

Minecraft

version: 1
service: minecraft
enabled: true

schedule:
  cron: "0 2,8,14,20 * * *"
  retry_after: 30m
  max_lateness: 2h

retention:
  keep_last: 8
  keep_daily: 14
  keep_weekly: 8
  keep_monthly: 12

compose:
  files: [compose.yaml]

consistency:
  mode: stop
  timeout: 120
  services: [minecraft]
  before: []
  after: []

sources:
  paths:
    - id: compose
      path: compose.yaml
    - id: env
      path: .env
      required: false
    - id: data
      path: ./data
      exclude:
        - logs/**
        - crash-reports/**
  volumes: []

Ghost + MySQL named volume

version: 1
service: ghost
enabled: true

schedule:
  cron: "0 3,15 * * *"
  retry_after: 30m
  max_lateness: 3h

retention:
  keep_last: 4
  keep_daily: 14
  keep_weekly: 8
  keep_monthly: 12

compose:
  files: [compose.yaml]

consistency:
  mode: stop
  timeout: 180
  services:
    - ghost
    - db

sources:
  paths:
    - id: compose
      path: compose.yaml
    - id: env
      path: .env
      required: false
    - id: content
      path: ./content

  volumes:
    - id: mysql-data
      compose_volume: db-data
資料庫較成熟的做法是使用 hooks 產生 SQL logical dump，再備份 dump。直接備份 raw MySQL volume 時，還原環境應維持相同資料庫產品與 major version。

16. 驗證並初始化 repository

列出 manifests

sudo backupctl list

完整驗證

sudo backupctl validate

驗證內容包括：

  • 全域 YAML 與每個 backup.yaml。
  • Compose 檔案與 docker compose config。
  • required paths。
  • Compose logical volume 與實際 Docker volume。
  • Cron、retention 與 consistency 欄位。
  • helper image、Restic、rclone、rsync。

第一次建立 repository

sudo backupctl init
init 只在 repository 尚不存在時執行。主機重建、重新連接既有 repository 時不要再次執行。

17. 執行首次備份

先測單一服務：

sudo backupctl backup minecraft

查看 snapshot：

sudo backupctl snapshots minecraft

再測其他服務：

sudo backupctl backup ghost

一次備份所有 enabled 服務：

sudo backupctl backup

查看排程狀態：

sudo backupctl status

18. 還原、選取與一次還原全部服務

不指定服務名稱時，會從 Restic repository 找出所有具有 service:<name> tag 的服務，開啟互動式選單。所有項目預設勾選：

sudo backupctl restore

操作鍵：

按鍵功能
↑／↓移動游標
Space切換目前服務
A全選或全部取消
Enter確認
Q／Esc取消

略過服務選單並還原全部服務

sudo backupctl restore --all

仍會進行最後確認。stdin 不是 TTY（例如自動化或 pipe）時必須明確加入 --yes，否則命令會拒絕執行。

還原、套用並啟動全部服務

sudo backupctl restore --all --apply --start --yes

也可只指定部分服務：

sudo backupctl restore minecraft ghost --apply

backup.yaml 的還原規則

  • 本機不存在 /srv/stacks/<service>/backup.yaml：直接從 snapshot 的 _meta/backup.yaml 還原，並依該 manifest 套用資料。
  • 本機已存在：互動詢問是否以 snapshot 中的 backup.yaml 覆蓋。
  • 可用 --restore-manifest 強制採用 snapshot manifest。
  • 可用 --keep-manifest 強制保留本機 manifest。

單純下載 snapshot 時，資料放在：

/var/lib/homelab-backup/restores/<service>/<timestamp>
第一次上線前至少實際執行一次 restore --apply 測試。成功上傳不等於能成功還原。
套用前會先驗證所有 required path、volume 與 Compose 檔案均存在於還原資料中，並停止該 Compose project 目前所有運行中的 services；任何必要來源缺失都會中止，不會以空目錄覆寫正式 volume。設定的 exclude 也會在還原時保留。

19. 啟用自動排程

只有在以下項目完成後才啟用：

  • validate 通過。
  • 每個服務至少成功備份一次。
  • 至少一個服務完成實際還原測試。
  • Restic 密碼已另外保存。
sudo systemctl enable --now homelab-backup.timer
sudo systemctl enable --now homelab-backup-maintenance.timer

確認：

systemctl list-timers 'homelab-backup*'
systemctl status homelab-backup.timer
systemctl status homelab-backup-maintenance.timer

每 5 分鐘 timer 執行：

backupctl run-due

它只備份當下到期的服務。

20. 監控與 repository 維護

查看服務執行紀錄

journalctl -u homelab-backup.service \
  -n 200 --no-pager

journalctl -u homelab-backup.service -f

檢查目前狀態

sudo backupctl status
sudo backupctl snapshots

預覽 retention

sudo backupctl maintenance --dry-run

正式 forget + prune

sudo backupctl maintenance

Repository check

sudo backupctl check

非正常中斷後移除 stale lock

sudo backupctl unlock

此命令只移除 Restic repository 判定為 stale 的 locks，不處理會隨 process 自動釋放的本機 flock。只有確認沒有其他 Restic process 執行時才能使用。

21. 災難復原必要材料

本專案以最少外部材料完成復原。真正必要的是：

材料用途
本專案程式碼從 https://github.com/flier268/homelab-backup.git 重新 clone。
configs/restic-password解密 Restic repository。
configs/rclone.conf存取 OneDrive；也可以重新授權取代。
configs/config.yaml提供 host ID、repository 路徑與本機工作路徑。
OneDrive Restic repository保存服務資料、歷史 snapshots、Compose 等來源檔，以及自動保存的 backup.yaml。

使用 ./backup-configs.sh 產生並保存前三個設定檔。使用 ./restore-configs.sh 放回對應系統路徑。

不需要另外備份 /srv/stacks 才能完成復原。用 private Git 保存 stacks 只會讓版本管理與人工查看更方便。
如果遺失 Restic 密碼，即使 OneDrive repository 還在，也無法解密資料。

22. 主機全毀後的最短重建流程

1. 重新取得專案

cd ~/Apps
git clone https://github.com/flier268/homelab-backup.git
cd homelab-backup

2. 準備 configs/

若設定已存入 private Git，clone 後應直接存在。若使用 ZIP，將 ZIP 解壓到專案目錄，最後必須是：

configs/restic-password
configs/rclone.conf
configs/config.yaml

3. 安裝工具

sudo ./install.sh

4. 放回必要設定

./restore-configs.sh

此腳本實際執行等效於：

sudo install -d -o root -g root -m 0700 \
  /etc/homelab-backup/rclone

sudo install -o root -g root -m 0600 \
  configs/restic-password \
  /etc/homelab-backup/restic-password

sudo install -o root -g root -m 0600 \
  configs/rclone.conf \
  /etc/homelab-backup/rclone/rclone.conf

sudo install -o root -g root -m 0600 \
  configs/config.yaml \
  /etc/homelab-backup/config.yaml

5. 測試 OneDrive 與 repository

sudo rclone lsd onedrive: \
  --config /etc/homelab-backup/rclone/rclone.conf

sudo backupctl snapshots

不要執行 backupctl init，因為這是連回原有 repository。

6. 還原全部服務

互動選取，預設全選：

sudo backupctl restore --apply --start

完全非互動：

sudo backupctl restore --all \
  --restore-manifest --apply --start --yes

本機沒有 backup.yaml 時，程式會先從 snapshot 還原它；之後依其中的 path 與 volume 定義還原各服務，不需要 bootstrap 命令。

7. 檢查服務

docker ps
cd /srv/stacks/ghost
docker compose ps
docker compose logs --tail=200

8. 最後才啟用排程

sudo systemctl enable --now homelab-backup.timer
sudo systemctl enable --now homelab-backup-maintenance.timer

部署完成條件

  • 兩台主機各自使用獨立 repository。
  • 所有 manifests 通過驗證。
  • 所有重要服務有成功 snapshot。
  • 至少完成一次實際 restore + apply。
  • configs/ 三個必要檔案已有安全的外部副本。
  • Restic 密碼、repository 設定與 OneDrive 授權資料可在主機外取得。
  • systemd timers 正常執行且 journal 無錯誤。
  • 舊備份方案在新方案還原驗證完成前不要刪除。
