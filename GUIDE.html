Docker Compose + Restic + OneDrive
每個服務自行宣告備份

從一台尚未部署本方案的 Linux 主機開始，完成工具安裝、OneDrive 授權、服務 manifest、首次備份、還原測試與 systemd 自動排程。

1. 目標與架構

服務仍由 Docker Compose 管理。 每個服務在自己的目錄放一份 backup.yaml，自行宣告排程、保留策略、備份來源與一致性方式。
/srv/stacks/minecraft/backup.yaml ─┐ /srv/stacks/palworld/backup.yaml ├─ backupctl 每 5 分鐘掃描 /srv/stacks/ghost/backup.yaml ┘ ↓ 停止服務或執行 hook ↓ 本機 staging ↓ Restic 加密、去重、快照 ↓ rclone → OneDrive repository
元件用途
Docker Compose執行 Minecraft、Palworld、Ghost 等服務。
backup.yaml每個服務自己的排程、保留策略與來源清單。
backupctl掃描 manifest、建立 staging、控制 Compose、執行 Restic。
Restic加密、去重、snapshot、forget、prune、check、restore。
rclone讓 Restic 使用 OneDrive 作為 repository backend。
systemd timer每 5 分鐘執行一次到期檢查；另有週期性維護。

2. 前置條件

  • Debian、Ubuntu 或 Linux Mint 類型系統。
  • 目前登入帳號具備 sudo 權限。
  • 兩台主機需要各自執行本手冊。
  • Microsoft OneDrive 帳號可正常登入。
  • 服務可以短暫停止，或能透過 hook 產生一致性快照。

下列命令用來確認系統：

cat /etc/os-release
uname -a
id
sudo -v

3. 安裝或確認 Docker Compose

先檢查：

docker --version
docker compose version

若兩者都能正常輸出，可以直接前往下一章。

Ubuntu／Linux Mint 安裝 Docker 官方套件

sudo apt update
sudo apt install -y ca-certificates curl

sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

. /etc/os-release

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
https://download.docker.com/linux/ubuntu \
${UBUNTU_CODENAME:-$VERSION_CODENAME} stable" \
| sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update
sudo apt install -y \
  docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin

sudo systemctl enable --now docker

讓目前帳號可操作 Docker

sudo usermod -aG docker "$USER"

登出後重新登入，再測試：

docker run --rm hello-world
docker compose version
加入 docker 群組等同授予高權限。這台電腦只有你使用時較容易接受，但仍不應把不受信任的人加入此群組。

4. 建立 Compose 服務目錄

Compose 專案統一放在：

/srv/stacks/<service>

建立 root 控制目錄。manifest、Compose files 與其父目錄不可由一般使用者寫入；需要修改時使用 sudoedit：

sudo install -d -o root -g root -m 755 /srv/stacks /srv/stacks/<service>
sudo chmod 755 /srv /srv/stacks
sudoedit /srv/stacks/<service>/backup.yaml
backup.yaml 可宣告由 root 執行的 consistency hooks，因此服務目錄與控制檔不能交給一般帳號或設成 group/world writable。服務目錄下的資料子樹則可由容器 UID/GID 擁有；不要對混有容器資料的目錄使用遞迴 chown。

建議結構：

/srv/stacks/
├── minecraft/
│   ├── compose.yaml
│   ├── .env
│   ├── backup.yaml
│   └── data/
├── palworld/
│   ├── compose.yaml
│   ├── .env
│   ├── backup.yaml
│   └── data/
└── ghost/
    ├── compose.yaml
    ├── .env
    ├── backup.yaml
    └── content/

一般權限：

chmod 644 compose.yaml backup.yaml
chmod 600 .env
Compose 不會自動載入隱含 .env，也不接受外部 COMPOSE_* 覆寫。需要環境插值時，必須在 backup.yaml 以 compose.env_file 明確指定；該檔案、所有 Compose YAML、Restic 密碼與 rclone 設定都必須是 root-owned、不可 group/world write 的普通檔案， 且不可為 symlink。

資料目錄先讓 container 建立或使用預設權限；只有遇到 permission denied 時，才依 image 的 UID/GID 個別調整。

5. 可選：用 Git 保存 /srv/stacks

可以把各服務的 compose.yaml、Dockerfile、Caddyfile、scripts 與其他非機密部署檔保存到 private Git，作為版本管理與額外副本。

這是加分項，不是本專案災難復原的必要條件，也不在本專案主要流程內。即使沒有另外備份 /srv/stacks，仍可從 Restic snapshot 取回服務資料、Compose 檔與自動保存的 backup.yaml。

不要把公開 Git 當成 secrets 儲存位置。真正的 .env、Restic 密碼與 rclone token 不應公開。

6. 從 GitHub 取得並安裝專案

若尚未安裝 Git：

sudo apt update
sudo apt install -y git
cd ~/Apps
git clone https://github.com/flier268/homelab-backup.git
cd homelab-backup
sudo ./install.sh

安裝器會：

  • 安裝 Python 3.10 以上版本、Python venv、rsync、Restic、rclone、age、OpenSSL。
  • 在 /usr/local/lib/homelab-backup/releases/ 建立包含 venv 與 application package 的完整 release。
  • 安裝 /usr/local/sbin/backupctl，並以原子 symlink 切換目前 release；失敗升級不會混用新舊 Python 檔案。
  • 建立全域設定、密碼、staging、restore、state 與 cache 路徑。
  • 為每個 release 建立並 smoke-test 專屬 named-volume rsync helper image，與 Python release 一起切換。
  • 安裝 systemd service 與 timer，但不立即啟用。
安裝期間不要關機。本專案不維護跨斷電安裝 journal；若安裝因斷電中止，重新執行 sudo ./install.sh，讓完整 release 重新發布。

解除安裝

停止排程並移除程式、systemd units、已安裝 releases 與 Docker helper images：

sudo ./uninstall.sh

預設保留 /etc/homelab-backup、/var/lib/homelab-backup 與 /var/cache/homelab-backup，方便稍後重新安裝。確定連本機設定、密碼、 state、staging、restore workspace 與 cache 都不需要時，才使用：

sudo ./uninstall.sh --purge
兩種模式都不會移除共用系統套件、遠端 Restic repository， 也不會刪除 /srv/stacks 或 /srv/data。

確認：

command -v backupctl
backupctl --help
restic version
rclone version
rsync --version

7. 設定 OneDrive

使用 root 的獨立 rclone 設定：

sudo rclone config \
  --config /etc/homelab-backup/rclone/rclone.conf

建立 remote 時：

  1. 選擇 n 建立新 remote。
  2. 名稱固定使用 onedrive。
  3. Storage 選 Microsoft OneDrive。
  4. 依提示完成 Microsoft OAuth。
  5. 選擇正確的 OneDrive drive。

完成後：

sudo chmod 600 /etc/homelab-backup/rclone/rclone.conf

sudo rclone lsd onedrive: \
  --config /etc/homelab-backup/rclone/rclone.conf

本方案不使用 rclone mount。Restic 直接使用：

rclone:onedrive:Backups/restic/server-a

8. 設定本機 repository

編輯：

sudo editor /etc/homelab-backup/config.yaml

第一台主機

version: 1
host_id: server-a
services_root: /srv/stacks
trusted_data_roots:
  - /srv/stacks
  - /srv/data

repository: rclone:onedrive:Backups/restic/server-a
password_file: /etc/homelab-backup/restic-password
rclone_config: /etc/homelab-backup/rclone/rclone.conf

staging_root: /var/lib/homelab-backup/staging
restore_root: /var/lib/homelab-backup/restores
state_root: /var/lib/homelab-backup/state
cache_root: /var/cache/homelab-backup/restic

lock_file: /run/homelab-backup/backupctl.lock
volume_helper_image: homelab/volume-rsync:1

rclone:
  bwlimit: ""

check:
  read_data_subset: 5%

volume_helper_image 是原始碼執行與舊版相容的預設值；正式安裝後，launcher 會使用目前 release 綁定且已 smoke-test 的 helper image，升級失敗不會覆寫舊 release 使用的 image。

全域設定只接受文件列出的欄位；rclone 與 check 必須是 mapping。錯誤型別會在載入時停止，不會延後到備份或 repository check 才崩潰。

第二台主機

只需改成：

host_id: server-b
repository: rclone:onedrive:Backups/restic/server-b
兩台主機可以共用同一個 OneDrive 帳號，但應使用不同 Restic repository。這樣 lock、prune、check 與故障範圍不會互相干擾。

9. 備份災難復原必要設定

完成 OneDrive 與全域設定後，在專案目錄執行：

cd ~/Apps/homelab-backup
sudo ./backup-configs.sh

此腳本只允許 root 執行。三個必要設定只會在 /run 的 root 專用暫存目錄封裝為 ZIP，離開該目錄前會先用貼入的 ssh-ed25519 或 ssh-rsa 公鑰經 age 加密。腳本會先確認 /run 確實位於 tmpfs。

功能邊界：這是全新專案，只支援目前的加密 archive，不提供舊明文設定、舊 archive 或 schema 遷移。全域設定、backup.yaml 與 snapshot inventory 都使用目前唯一的 version: 1。

接著只能選一種保存方式：

  1. Git：只建立並提交 configs/homelab-backup-configs.zip.age。腳本會先顯示 configured remotes；輸入完整的 PRIVATE 後才提交並詢問是否 git push。
  2. 加密檔：建立 timestamped .zip.age。
私鑰必須另外保存在 Bitwarden 或離線安全媒體。沒有對應私鑰便無法還原；加密檔仍建議只放在嚴格控管的 private repository。

腳本只以 root 讀取系統設定；選擇 Git 後，所有 Git 指令、hook 與 filter 都會降權成原本執行 sudo 的非 root 使用者。無法辨識非 root 使用者時會拒絕 Git 選項。

解密並放回設定

cd ~/Apps/homelab-backup
sudo ./restore-configs.sh

依提示從 Bitwarden 貼入完整 SSH 私鑰，再按 Ctrl-D。也可在命令尾端指定另一個 .zip.age archive。

還原腳本會將檔案安裝到：

/etc/homelab-backup/restic-password
/etc/homelab-backup/rclone/rclone.conf
/etc/homelab-backup/config.yaml

還原前會確認 ZIP 恰好含三個預期檔案。三個設定會先完整準備，安裝失敗時全部回滾，不留下新舊版本混合狀態。

設定還原期間不要關機。一般錯誤會在同一程序內回滾；斷電不在原子性保證內，重新開機後請再次執行相同的 restore-configs.sh。

換 SSH KEY

sudo ./backup-configs.sh --rotate configs/homelab-backup-configs.zip.age

先貼入舊私鑰並按 Ctrl-D，再貼入新公鑰。archive 必須由執行 sudo 的非 root 使用者擁有；原檔讀取與最後替換都以該使用者權限執行。腳本只在完整解密、內容驗證及重新加密成功後原子替換舊檔。

功能邊界：本專案不管理 SSH agent、金鑰生命週期或遠端祕密服務。私鑰只從標準輸入直接交給 age，不寫入參數、環境變數或一般磁碟；換 KEY 後需自行更新 Bitwarden 備援。

10. 每個服務的 backup.yaml

每個服務在自己的 Compose 目錄放一份 manifest：

backup.yaml 由備份程式自動保存到每份 snapshot 的 _meta/backup.yaml，不需要再把它寫進 sources.paths。
目錄名稱必須與 service 完全相同；manifest、compose、sources 與各 source entry 的未知欄位會被拒絕，避免拼字錯誤造成靜默漏備份。
/srv/stacks/minecraft/backup.yaml

完整結構：

version: 1
service: minecraft
enabled: true

schedule:
  cron: "0 2,8,14,20 * * *"
  retry_after: 30m
  max_lateness: 2h

retention:
  keep_last: 8
  keep_daily: 14
  keep_weekly: 8
  keep_monthly: 12
  keep_yearly: 3

compose:
  files:
    - compose.yaml
  env_file: .env

consistency:
  mode: stop
  timeout: 120
  before: []
  after: []

actions:
  before: []
  finally: []
  on_success: []
  on_failure: []

sources:
  paths:
    - id: compose
      path: compose.yaml

    - id: env
      path: .env
      required: false

    - id: data
      path: ./data
      exclude:
        - logs/**
        - crash-reports/**

  volumes: []
若服務原本依賴 Compose 自動尋找 .env，升級後 必須加入 compose.env_file；未設定時會明確使用空環境檔。

11. 一致性模式

模式行為用途
stop停止整個 Compose project 中原本運行的 services → 動態 preflight → 單次同步與 volume 備份 → 重啟原本運行的 services。Minecraft、Palworld、持續寫入檔案、raw database volume。
hooks執行 before hook → 動態 preflight → 同步 → 執行 after hook；before hook 可建立或更新 sources.paths payload artifact，且必須確實停止或隔離所有 target writer。所有 required Docker named volumes 必須在靜態 preflight 前已存在，不支援由 hook 動態建立。SQL dump、RCON save、應用內建 snapshot。
external不控制服務，但要求所有 Docker writer 已由外部停止，再同步至 staging。由其他編排器靜止的服務、唯讀資料。
live不停止 writer，直接同步至 staging；偵測到 writer 時警告並在 inventory 標成 best-effort。可接受 fuzzy copy 的可重建內容。
snapshot來源 path 本身是 Btrfs subvolume 時建立唯讀 snapshot；普通 path 與 named volume 直接同步。未受 snapshot 保護且仍有 writer 的來源標成 best-effort。不停機的 Palworld、Minecraft 與其他檔案型服務。

stop 範例

consistency:
  mode: stop
  timeout: 180

hooks 範例

consistency:
  mode: hooks
  before:
    - ./scripts/dump-database.sh
  after: []

備份交易 action

staging copy、Docker named volume copy、Restic 備份與 retention 固定以 root 執行，以完整保存來源的數字 UID/GID、mode、ACL、xattr 與 hardlink。還原也以 root 套用 snapshot 內的原始 metadata；完成後的資料不會一律變成 root:root，因此原本以指定 UID/GID 執行的容器仍可存取。

actions.before 與 actions.finally 適用所有 mode，命令以 argv 執行、不經 shell。非零退出、找不到程式或逾時時，required: true 會中止整份備份；false 會警告、繼續並把整體保證降為 best-effort。Finally 位於 staging 交易的最外層：會在 stop 重啟、hook 恢復或 snapshot 清理之後執行，before 或 staging 失敗時也會嘗試執行；finally 錯誤不會覆蓋既有主要錯誤。

actions:
  before:
    - name: save-palworld
      command:
        - /usr/bin/docker
        - compose
        - exec
        - -T
        - palworld
        - rcon-cli
        - Save
      timeout: 30
      required: true
      run_as: root
  finally:
    - name: notify-backup-complete
      command: [/usr/bin/docker, exec, palworld, rcon-cli, Broadcast, BackupComplete]
      timeout: 30
      required: true
      run_as: root

每個 action 都必須明確設定 run_as，可使用本機帳號名稱或加引號的 Docker 風格 "UID:GID"；數字身分不需要 passwd entry。相對 executable 會被拒絕。run_as: root 或 run_as: "0:0" 的 executable 與父目錄必須由 root 擁有且不可由 group/other 寫入，也不可為 symlink。Docker socket 權限實質接近 root，能直接使用 HTTP/RCON endpoint 時不要把低權限帳號加入 docker group。

actions.on_success 在 Restic snapshot 成功提交後執行；actions.on_failure 在 staging、Restic、on-success 或 state 失敗時執行。Failure action 會收到 BACKUPCTL_FAILURE_PHASE、BACKUPCTL_FAILURE_TYPE、BACKUPCTL_FAILURE_REASON、BACKUPCTL_FAILURE_SERVICE 與 BACKUPCTL_FAILURE_SECONDARY。不會提供命令輸出或 argv；若命令使用 docker exec，需自行明確把需要的環境變數轉交給容器。

actions:
  on_success:
    - name: notify-success
      command: [/usr/local/bin/notify-backup, success]
      timeout: 15
      required: false
      run_as: root
  on_failure:
    - name: notify-failure
      command: [/usr/local/bin/notify-backup-failure]
      timeout: 15
      required: false
      run_as: root
Restic 已成功後，required on-success action 失敗仍會觸發 on-failure 並讓命令回傳失敗；但已提交的 snapshot 不可能撤回，因此 state 仍記錄該次備份成功，排程不會把它當成可重試的備份失敗。

HTTP GET／POST 可把 curl 與每個參數分別放入 command 陣列。密碼與 token 不要放進 argv 或 manifest；使用 root-only curl config、Docker secret 或 RCON 工具的密碼檔。

none 已移除；原本需要拒絕 writer 的設定改用 external，明確接受線上 fuzzy copy 才使用 live。

snapshot 的暫存唯讀 subvolume 固定位於來源所屬 trusted root 的 .homelab-backup-snapshots root-only 0700 workspace，並與來源位於同一 filesystem。建立前會以 FD 釘住來源，因此 container-owned parent 隨後被 rename 或替換不會改變實際 snapshot 來源。內部 journal 以 creating、ready、deleting 記錄 filesystem UUID、source UUID／subvolume ID、snapshot UUID／subvolume ID、parent UUID 與 path；每次備份的 action 前及非 dry-run maintenance 都會自動接管 crash 遺留項目，包含已停用或已移除服務。live source 已消失、rename 或重建時，舊 snapshot 仍依自身完整 identity 回收；任一 identity 不吻合則拒絕自動刪除。來源含 nested subvolume 時必須拆成各自的 source。多個獨立 subvolume 會依序完成全部 snapshot 後才開始 staging，但不宣稱它們具有跨 subvolume 的同時點原子性。

Inventory 的每個 path／volume 會記錄 capture_method；頂層 consistency 則記錄實際 mode、整體 guarantee、staging 前後的 optional action failures 與偵測到的 writer container IDs。Restic 結果之後才執行的 success／failure action 不會回寫已提交的 snapshot inventory。命令輸出、敏感 argv 與暫存 snapshot path 不會寫入 inventory。

不確定時先使用 stop。它較保守，也比較容易實際驗證還原結果。

12. Bind mount 與 named volume

Bind mount

Compose：

services:
  minecraft:
    volumes:
      - ./data:/data

Manifest：

sources:
  paths:
    - id: data
      path: ./data
      exclude:
        - logs/**
        - crash-reports/**

路徑來源的安全邊界

  • 每個 path 必須嚴格位於唯一的 trusted root 之下，不能等於 trusted root。trusted root 與其控制路徑必須由 root 擁有、不可 group/world write 且不是 symlink；其下的資料樹可完全由容器 UID/GID 擁有，也可選取其中任意深層路徑。程式從 trusted root 的固定 descriptor 逐層開啟來源與還原目標，中間 symlink 會被拒絕，可寫祖先被換名也不會把 root 寫入導向外部。
  • 備份會自動在 snapshot inventory 記錄所選路徑祖先的數字 UID/GID 與 mode。全新 rebuild 缺少資料祖先時會依原值重建，不會猜成 root:root 或 1000:1000，manifest 不需增加欄位。沒有這項 metadata 的舊快照仍可還原到祖先已存在的部署，但不會在全新 rebuild 時猜測權限。
  • sources.paths 指到 symlink 時，只保存 link 物件與其目標字串；目標是檔案、目錄或不存在（dangling）都不會被跟隨。
  • 真實目錄內的 symlink 同樣由 rsync 以 link 保存。還原時會替換目標位置的 leaf 物件，不會沿著現場 symlink 寫入其 referent。
  • 只支援 ext4、xfs 與 Btrfs。trusted root 可為核准 filesystem mount 或已掛載 Btrfs subvolume，但其下不允許其他 mount、bind mount或 nested Btrfs subvolume。
  • payload 支援普通檔案、目錄、symlink、ACL、xattr 與 payload 內 hardlink；FIFO、socket與 device node拒絕。Btrfs subvolume identity、snapshot、reflink、compression與 CoW 屬性不在保存承諾內。
  • 本版不支援 remote/rootless Docker、ZFS、ext2/3、FUSE、NFS、autofs、follow-link 或 cross-mount。
  • staging 與 restore workspace 套用相同 filesystem allowlist、control ownership 與 nested mount 驗證。

Compose named volume

Compose：

services:
  db:
    volumes:
      - db-data:/var/lib/mysql

volumes:
  db-data:

Manifest：

sources:
  volumes:
    - id: database
      compose_volume: db-data

compose_volume 填 Compose 頂層邏輯 key，不是實際 Docker volume 名稱。工具會透過 docker compose config --format json 解析。

外部 volume

sources:
  volumes:
    - id: shared-data
      name: company_shared_data

name 只接受 Docker 管理的 named volume 名稱，不接受 /etc、相對 host path 或 mount option。工具以 --mount type=volume 明確掛載，絕不把 volume name 推測成 bind mount。

Anonymous volume

以下形式沒有穩定名稱，不適合宣告式備份：

volumes:
  - /data

應改成 named volume：

services:
  app:
    volumes:
      - app-data:/data

volumes:
  app-data:

13. Cron 排程

格式固定為標準 5 欄：

分鐘 小時 日期 月份 星期
表達式意義
0 2 * * *每天 02:00
0 2,8,14,20 * * *每天 02:00、08:00、14:00、20:00
*/15 * * * *每 15 分鐘
0 */6 * * *每 6 小時整點
30 3 * * mon-fri週一到週五 03:30
0 4 1 * *每月 1 日 04:00

排程器每 5 分鐘掃描，因此實際開始時間可能比 cron 時點晚數分鐘。

cron 時點由 CronSim 依 Debian cron 語意計算。夏令時間向前跳時，落在略過區間的固定時刻會在跳時後立即執行；向後回撥時，同一固定時刻不會重複執行。

備份超過 5 分鐘

  • systemd oneshot 不會同時再啟動第二份相同 service。
  • 全域 flock 阻止手動備份、排程、還原與維護重疊。
  • 多個服務同時到期時依序執行，不平行寫入 repository。
  • 執行期間經過的 cron 時點不建立工作佇列。
例如 02:00 開始，備份到 08:20，而 cron 也包含 08:00：08:00 不會另開第二份；完成後下一個有效時點是 14:00。
高頻排程同理：若 */5 * * * * 的 12:00 備份到 12:17 才完成，12:05、12:10、12:15 全部跳過；下一次是 12:20，不會在完成後立即補跑。

失敗重試

schedule:
  cron: "0 */6 * * *"
  retry_after: 30m
  max_lateness: 2h

失敗後至少等待 30 分鐘再重試；若該 cron 時點已遲到超過 2 小時，放棄該次，等待下一時點。

14. 每個服務的保留策略

retention:
  keep_last: 8
  keep_daily: 14
  keep_weekly: 8
  keep_monthly: 12
  keep_yearly: 3
欄位意義
keep_last保留最新 N 份。
keep_hourly最近 N 個有 snapshot 的小時，各留一份。
keep_daily最近 N 個有 snapshot 的日期，各留一份。
keep_weekly最近 N 個有 snapshot 的週，各留一份。
keep_monthly最近 N 個有 snapshot 的月份，各留一份。
keep_yearly最近 N 個有 snapshot 的年份，各留一份。
多個 keep-* 條件是聯集，不是總數相加，也不是絕對最大數量。只要符合任一規則就保留。若要嚴格只留最新 30 份，只設定 keep_last: 30。

16. 驗證並初始化 repository

列出 manifests

sudo backupctl list

完整驗證

sudo backupctl validate

驗證內容包括：

  • 全域 YAML 與每個 backup.yaml。
  • Compose 檔案與 docker compose config。
  • required paths。
  • Compose logical volume 與實際 Docker volume。
  • Cron、retention 與 consistency 欄位。
  • helper image、Restic、rclone、rsync。

第一次建立 repository

sudo backupctl init
init 只在 repository 尚不存在時執行。主機重建、重新連接既有 repository 時不要再次執行。

17. 執行首次備份

先測單一服務：

sudo backupctl backup minecraft

查看 snapshot：

sudo backupctl snapshots minecraft

刪除指定 snapshot 時會先確認 ID 屬於目前 host 與指定服務。互動模式會再次確認；非互動模式必須加 --yes。預設未引用資料會留待下次 maintenance 回收；加 --prune 可立即回收：

sudo backupctl delete-snapshot minecraft 01234567
sudo backupctl delete-snapshot minecraft 01234567 --prune --yes

再測其他服務：

sudo backupctl backup ghost
sudo backupctl backup palworld

一次備份所有 enabled 服務：

sudo backupctl backup

批次備份會逐 manifest 隔離錯誤；某一服務失敗時會跳過整份 manifest、清除其半成品 staging，且只有全部來源 staging 完成後才提交 Restic snapshot，因此不會產生該服務的部分備份。其餘 manifest 仍會繼續，最後統一列出失敗摘要並回傳非零狀態。

來源靜止後會保守估算 path 與 Docker volume 的 staging 大小，並要求所在 filesystem 至少保留 1 GiB。空間不足時預設中止；只有人工操作可明確使用：

sudo backupctl backup minecraft --allow-low-space
run-due 不會自動略過空間保護。即使人工放行， 實際複製遇到 ENOSPC 仍會失敗並清除該服務的半成品 staging。

查看排程狀態：

sudo backupctl status

18. 還原、選取與一次還原全部服務

不指定服務名稱時，會從 Restic repository 找出所有具有 service:<name> tag 的服務，開啟互動式選單。所有項目預設勾選：

sudo backupctl restore

操作鍵：

按鍵功能
↑／↓移動游標
Space切換目前服務
A全選或全部取消
Enter確認
Q／Esc取消

略過服務選單並還原全部服務

sudo backupctl restore --all

仍會進行最後確認。安全政策規定 stdin 不是 TTY（例如自動化或 pipe）時，所有 restore（包含只下載、不使用 --apply）都必須明確加入 --yes，否則命令會拒絕執行。

還原、套用並啟動全部服務

sudo backupctl restore --all --apply --start --yes

也可只指定部分服務：

sudo backupctl restore minecraft ghost --apply

批次還原同樣逐服務隔離錯誤；某一服務下載或套用失敗時，不會跳過後續服務，最後會列出成功結果與失敗摘要。

指定歷史 snapshot ID 時只允許單一服務。程式會先用目前 host 與 service:<name> tag 解析並驗證完整 ID，再開始下載：

sudo backupctl restore minecraft --snapshot 01234567 --apply

下載前會以 Restic restore-size 估算展開大小，並要求至少保留 1 GiB。互動模式可再次確認；非互動模式必須另外指定 --allow-low-space，而 --yes 本身不會略過空間保護。

backup.yaml 的還原規則

  • 現有部署：本機 manifest 與所有 Compose files 都存在；snapshot inventory 必須與本機 project name、path/source declarations、volume mapping、volume 名稱及 labels 相符後才可覆寫。Compose service 清單只供診斷，新增或移除 service 不會單獨阻止舊 snapshot 還原。
  • 全新重建：manifest、Compose files、所有 present path與 named volume 都必須不存在，且 snapshot 中必須包含所有 required Compose file path source；資料完成後才發布 Compose files與 manifest。目錄 path 會先在私有暫存目錄完整準備再原子發布。若發布完成前失敗，會嘗試移除本次新建的 path 與 volume，使同一還原可以重跑；rollback 自身失敗時會保留原始錯誤並另外回報 cleanup 錯誤。若 path 在發布後被其他程序修改，或新建祖先已出現其他內容，則保留該路徑並警告，避免刪除外部寫入。
  • 只有部分設定或 target 存在、project identity 不明、或有其他 container 引用 target時，一律拒絕，不猜測身份也不覆寫。
  • --restore-manifest 只允許更新與本機安全投影一致的 snapshot manifest。
  • 可用 --keep-manifest 強制保留本機 manifest。
資料還原期間不要關機。一般錯誤會在目前程序內嘗試 rollback；斷電不在交易保證內。重新開機後可再次執行還原；全新重建若留下 partial target，先依錯誤列出的 path／volume 確認並移除，再重跑。

單純下載 snapshot 時，資料放在：

/var/lib/homelab-backup/restores/<service>/<timestamp>

restore --apply 成功後會自動刪除該次下載副本；套用失敗或只下載時會保留。可手動刪除一個、多個，或批量刪除全部保留副本：

sudo backupctl cleanup-restores minecraft/20260717-120000-000000001 --yes
sudo backupctl cleanup-restores \
  minecraft/20260717-120000-000000001 \
  ghost/20260717-120500-000000002 --yes
sudo backupctl cleanup-restores --all --yes
第一次上線前至少實際執行一次 restore --apply 測試。成功上傳不等於能成功還原。
套用前會先驗證所有 required path、volume 與 Compose 檔案均存在於還原資料中，並停止該 Compose project 目前所有運行中的 services；任何必要來源缺失都會中止，不會以空目錄覆寫正式 volume。設定的 exclude 也會在還原時保留。
操作期間不存在未受控 writer 是 operator obligation。程式會檢查已知 Docker containers，但無法證明主機上沒有共用 UID 程序、排程或其他 privileged process。

19. 啟用自動排程

只有在以下項目完成後才啟用：

  • validate 通過。
  • 每個服務至少成功備份一次。
  • 至少一個服務完成實際還原測試。
  • Restic 密碼已另外保存。
sudo systemctl enable --now homelab-backup.timer
sudo systemctl enable --now homelab-backup-maintenance.timer

確認：

systemctl list-timers 'homelab-backup*'
systemctl status homelab-backup.timer
systemctl status homelab-backup-maintenance.timer

每 5 分鐘 timer 執行：

backupctl run-due

它只備份當下到期的服務。

材料用途

22. 主機全毀後的最短重建流程

1. 重新取得專案

cd ~/Apps
git clone https://github.com/flier268/homelab-backup.git
cd homelab-backup

2. 準備加密設定 archive 與私鑰

若 archive 已存入 private Git，clone 後會位於 configs/homelab-backup-configs.zip.age。另從 Bitwarden 準備對應 SSH 私鑰。

3. 安裝工具

sudo ./install.sh

4. 放回必要設定

sudo ./restore-configs.sh

依提示貼入完整私鑰並按 Ctrl-D。使用外部 archive 時，將路徑附在命令後；內容驗證成功後才以 root、mode 0600 安裝三個設定，任一步失敗會回滾整組設定。

5. 測試 OneDrive 與 repository

sudo rclone lsd onedrive: \
  --config /etc/homelab-backup/rclone/rclone.conf

sudo backupctl snapshots

不要執行 backupctl init，因為這是連回原有 repository。

6. 還原全部服務

互動選取，預設全選：

sudo backupctl restore --apply --start

完全非互動：

sudo backupctl restore --all \
  --restore-manifest --apply --start --yes

本機沒有 backup.yaml 時，只有在 manifest、所有 Compose files、present paths、named volumes 與相關 container references 全部不存在的全新重建模式下才會套用；任何混合狀態都會拒絕。

7. 檢查服務

docker ps
cd /srv/stacks/ghost
docker compose ps
docker compose logs --tail=200

8. 最後才啟用排程

sudo systemctl enable --now homelab-backup.timer
sudo systemctl enable --now homelab-backup-maintenance.timer

部署完成條件

  • 兩台主機各自使用獨立 repository。
  • 所有 manifests 通過驗證。
  • 所有重要服務有成功 snapshot。
  • 至少完成一次實際 restore + apply。
  • configs/ 三個必要檔案已有安全的外部副本。
  • Restic 密碼、repository 設定與 OneDrive 授權資料可在主機外取得。
  • systemd timers 正常執行且 journal 無錯誤。
